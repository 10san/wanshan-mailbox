## 1. 产品背景与定位

### 1.1 市场背景

- 2023 年中国匿名社交软件市场总用户数持续增长，35 岁以下用户占比超过 75% [1]。
- 当前市场上树洞类产品以微信公众号形态为主（如红尘树洞、旧纸树洞等），独立 Web 端产品稀缺，存在差异化空间。
- 现有匿名倾诉平台普遍存在隐私泄露、收费套路、服务质量不稳定三大痛点 [2]。
- 开源社区中已有成熟参考项目（Treehole Project、XAUFEHole 等），但均缺少完整的管理后台和内容治理能力。

### 1.2 产品定位

**晚山信箱** 是一款 **面向年轻群体的匿名情感树洞 Web 应用**。定位为：

- 零门槛、纯匿名的情感倾诉空间
- 轻量化、治愈系的内容社区
- 开源免费、自部署的经济型方案

> "晚山" 取自 "日暮苍山远"，寓意在一天的疲惫之后，有一个可以安放情绪的地方。

### 1.3 核心差异化

| 对比维度 | 市场主流（微信公众号树洞） | 晚山信箱 |
|---------|----------------------|---------|
| 产品形态 | 公众号内嵌 H5 | 独立 Web 应用，后续拓展小程序 |
| 商业模式 | 免费+付费深度服务 | V1 纯免费，无商业变现 |
| 技术开放性 | 闭源 | 全栈开源（MIT License） |
| 内容治理 | 平台单方面管控 | 社区举报+管理员审核双机制 |
| 数据控制 | 用户数据存于第三方 | 自部署，数据自主可控 |

## 2. 产品目标

### 2.1 V1 版本核心目标

| 目标类型 | 目标描述 | 衡量指标 |
|---------|---------|---------|
| **功能目标** | 实现完整的匿名发帖-浏览-互动闭环 | 核心功能可用率 100% |
| **体验目标** | 用户在 30 秒内完成首次发帖 | 发帖流程 ≤ 3 步 |
| **安全目标** | 敏感词过滤准确率 ≥ 95%，违规内容 30 分钟内可处理 | 审核时效 ≤ 30min |
| **技术目标** | 前后端分离架构，API 可复用于后续小程序 | 接口全部 RESTful，带 Swagger 文档 |
| **成本目标** | 单服务器（2C4G）可承载 500+ DAU | 服务器成本 ≤ 100 元/月 |

### 2.2 非目标（V1 不做）

- ❌ 不做用户注册/登录（纯匿名）
- ❌ 不做私信/一对一聊天
- ❌ 不做语音/视频内容
- ❌ 不做付费功能
- ❌ 不做 AI 情绪分析/智能匹配
- ❌ 不做小程序端（架构预留）

## 6. 核心流程说明

### 6.1 用户发帖流程

```
[用户进入首页] → [点击"写心事"] → [输入文字内容]
    → [选择情绪标签（可选）] → [设置删除密码（可选）]
    → [点击发布] → [敏感词检测]
        ├─ 通过 → [发布成功，跳转至新帖子]
        └─ 不通过 → [提示含敏感词，返回修改]
```

### 6.2 用户浏览与互动流程

```
[用户进入首页] → [浏览帖子卡片流]
    ├─ [点击帖子卡片] → [进入详情页]
    │   ├─ [查看完整内容]
    │   ├─ [查看评论区]
    │   ├─ [点赞/取消点赞]
    │   ├─ [点击"抱抱"/"同感"]
    │   ├─ [发表评论] → [敏感词检测] → [评论成功]
    │   └─ [举报]
    └─ [下拉加载更多]
```

### 6.3 管理员处理举报流程

```
[管理员登录后台] → [查看举报列表]
    → [点击查看被举报内容]
        ├─ 确认违规 → [删除内容] → [标记举报"已处理"]
        └─ 不违规 → [标记举报"已驳回"]
```

## 7. 功能需求详述（EARS 格式）

### 7.1 匿名发帖

#### FR-01-01（Ubiquitous）
**The system shall** 允许任何访问者在不注册、不登录的情况下发布树洞帖子。

#### FR-01-02（Event-driven）
**When** 用户点击"发布"按钮，**the system shall** 对帖子正文进行敏感词检测，检测通过后创建帖子并分配唯一 ID，帖子默认状态为"已发布"。

#### FR-01-03（Event-driven）
**When** 用户提交的帖子内容命中敏感词库，**the system shall** 拒绝发布并提示"内容包含不当信息，请修改后重新发布"，不告知具体敏感词。

#### FR-01-04（State-driven）
**While** 帖子处于"已发布"状态，**the system shall** 在首页帖子列表和详情页中展示该帖子。

#### FR-01-05（State-driven）
**While** 帖子被管理员标记为"已删除"，**the system shall** 不在任何公开页面展示该帖子，访问时返回"内容不存在"。

#### FR-01-06（Optional）— 评审确认：删除密码为可选字段
**Where** 用户在发帖时填写了删除密码，**the system shall** 将密码使用 BCrypt 哈希存储，并允许用户凭该密码删除帖子。
**Where** 用户未填写删除密码，**the system shall** 允许发布，但该帖子仅能由管理员删除，用户无法自行删除。

#### FR-01-07（Optional）
**Where** 用户选择了情绪标签，**the system shall** 将标签与帖子关联，用户可通过标签筛选帖子。

#### FR-01-08（Unwanted）
**If** 帖子正文为空或仅含空白字符，**then the system shall** 拒绝发布并提示"请输入内容"。

#### FR-01-09（Unwanted）
**If** 帖子正文字数超过 2000 字，**then the system shall** 拒绝发布并提示"内容超出字数限制（2000字）"。

### 7.2 帖子列表

#### FR-02-01（Ubiquitous）
**The system shall** 在首页以卡片流形式展示帖子列表，默认按发布时间倒序排列，每页加载 20 条。

#### FR-02-02（Event-driven）
**When** 用户滚动到列表底部，**the system shall** 自动加载下一页数据，直至无更多内容时显示"没有更多了"。

#### FR-02-03（Event-driven）
**When** 用户点击标签筛选按钮，**the system shall** 仅展示带有该标签的帖子。

#### FR-02-04（Event-driven）
**When** 用户输入搜索关键词并提交，**the system shall** 对帖子标题和正文进行全文检索，返回匹配结果。

#### FR-02-05（Ubiquitous）
**The system shall** 在每张帖子卡片上展示：匿名昵称、发布时间（相对时间）、正文摘要（前 120 字）、标签、点赞数、评论数。

### 7.3 帖子详情与评论

#### FR-03-01（Event-driven）
**When** 用户点击帖子卡片，**the system shall** 跳转至帖子详情页，展示完整内容、标签、发布时间、点赞数、"抱抱"数、"同感"数。

#### FR-03-02（Ubiquitous）
**The system shall** 在帖子详情页下方展示评论区，评论按发布时间正序排列。

#### FR-03-03（Event-driven）
**When** 用户在评论区提交评论，**the system shall** 进行敏感词检测，通过后发布评论，评论内容限制 500 字以内。

#### FR-03-04（Unwanted）
**If** 评论内容为空或命中敏感词，**then the system shall** 拒绝发布并给出相应提示。

### 7.4 点赞与互动

#### FR-04-01（Event-driven）
**When** 用户点击帖子的点赞按钮，**the system shall** 记录点赞（基于 IP+Cookie 去重），点赞数 +1；若已点赞则取消点赞，点赞数 -1。

#### FR-04-02（Event-driven）
**When** 用户点击评论的点赞按钮，**the system shall** 同上逻辑处理评论点赞。

#### FR-04-03（Event-driven）
**When** 用户点击"抱抱"或"同感"按钮，**the system shall** 记录对应互动计数（同样 IP+Cookie 去重）。

### 7.5 匿名身份

#### FR-05-01（Ubiquitous）
**The system shall** 为每个独立的浏览器会话（基于 Cookie）分配一个系统生成的匿名昵称和头像颜色。

#### FR-05-02（Ubiquitous）
**The system shall** 匿名昵称格式为「随机形容词 + 随机名词」，例如"安静的考拉""勇敢的向日葵"。同一设备同一 Cookie 下保持昵称不变。

#### FR-05-03（Ubiquitous）
**The system shall** 不公开用户的 IP 地址、浏览器指纹等任何可追溯真实身份的信息。

### 7.6 内容删除 — 评审确认：采用假删除（状态标记），不做物理删除

#### FR-06-01（Event-driven）
**When** 用户在帖子详情页点击"删除"，**the system shall** 弹出密码输入框。若帖子发布时未设置删除密码，则提示"该帖子未设置删除密码，如需删除请联系管理员"。

#### FR-06-02（Event-driven）
**When** 用户输入正确的删除密码，**the system shall** 将帖子状态更新为 `status=2`（用户删除），内容不再公开显示，但数据保留在数据库中。

#### FR-06-03（Ubiquitous）
**The system shall** 所有删除操作均为软删除（状态标记），不物理删除数据。状态枚举：`1`=正常，`2`=用户删除，`3`=管理员删除。

#### FR-06-04（Unwanted）
**If** 帖子发布时未设置删除密码，**then the system shall** 不允许用户删除（仅管理员可操作）。

#### FR-06-05（Unwanted）
**If** 输入错误删除密码超过 5 次，**then the system shall** 拒绝当次会话的删除尝试，提示"密码错误次数过多，请稍后再试"。

### 7.7 举报功能

#### FR-07-01（Event-driven）
**When** 用户点击帖子或评论的"举报"按钮，**the system shall** 展示举报理由选项（色情低俗、暴力欺凌、虚假信息、侵犯隐私、其他）。

#### FR-07-02（Event-driven）
**When** 用户选择举报理由并提交，**the system shall** 创建一条举报记录，状态为"待处理"，并向管理后台推送通知。

#### FR-07-03（Ubiquitous）
**The system shall** 对同一用户（IP+Cookie）对同一内容的重复举报进行去重，提示"您已举报过该内容"。

### 7.8 图片上传

#### FR-08-01（Event-driven）
**When** 用户在发帖时选择上传图片，**the system shall** 将图片上传至 MinIO 对象存储，并返回访问 URL 关联到帖子。

#### FR-08-02（Ubiquitous）
**The system shall** 限制单张图片大小为 5MB 以内，格式仅支持 JPG/PNG/WebP。

#### FR-08-03（Ubiquitous）
**The system shall** 对上传图片自动压缩至宽度 1200px，生成缩略图用于列表展示。

#### FR-08-04（Unwanted）
**If** 图片上传失败（MinIO 不可用或网络异常），**then the system shall** 提示"图片上传失败，请稍后重试"，但不影响文字内容发布。

### 7.9 敏感词过滤

#### FR-09-01（Ubiquitous）
**The system shall** 对所有用户提交的文本内容（帖子正文、评论）进行敏感词检测。

#### FR-09-02（Ubiquitous）
**The system shall** 敏感词库支持精确匹配和正则匹配两种模式。

#### FR-09-03（Event-driven）
**When** 管理员在后台新增或删除敏感词，**the system shall** 实时生效，无需重启服务。

### 7.10 管理后台 - 内容管理

#### FR-10-01（Ubiquitous）
**The system shall** 提供 Web 管理后台，仅管理员通过账号密码登录后访问。

#### FR-10-02（Event-driven）
**When** 管理员在内容列表查看帖子，**the system shall** 展示：匿名昵称、正文摘要、发布时间、标签、点赞数、评论数、状态。

#### FR-10-03（Event-driven）
**When** 管理员点击删除某条帖子或评论，**the system shall** 弹出二次确认，确认后执行软删除，内容状态变更为"管理员删除"。

#### FR-10-04（Ubiquitous）
**The system shall** 管理员支持按状态（正常/已删除）、时间范围、关键词筛选内容。

### 7.11 管理后台 - 举报处理

#### FR-11-01（Event-driven）
**When** 管理员进入举报处理页面，**the system shall** 展示所有举报记录，按"待处理"优先、举报时间倒序排列。

#### FR-11-02（Event-driven）
**When** 管理员点击"处理"，**the system shall** 展示被举报内容详情和举报理由，管理员可选择"删除内容"或"驳回举报"。

#### FR-11-03（Event-driven）
**When** 管理员选择"删除内容"，**the system shall** 删除对应内容并将举报状态更新为"已处理（删除）"。

#### FR-11-04（Event-driven）
**When** 管理员选择"驳回举报"，**the system shall** 保留内容并将举报状态更新为"已驳回"。

### 7.12 管理后台 - IP 封禁 — 评审新增：恶意灌水治理

#### FR-12-01（Event-driven）
**When** 管理员在内容管理或举报处理中发现恶意灌水行为，**the system shall** 支持将对应 IP 哈希加入封禁名单。

#### FR-12-02（State-driven）
**While** IP 哈希处于封禁状态，**the system shall** 拒绝该 IP 的发帖和评论请求，返回"您的操作已被限制"。

#### FR-12-03（Ubiquitous）
**The system shall** IP 封禁仅基于 IP 哈希（SHA-256），不存储原始 IP 地址。

#### FR-12-04（Event-driven）
**When** 管理员解除封禁，**the system shall** 立即恢复该 IP 的发帖和评论权限。

### 7.13 管理后台 - 数据看板

#### FR-13-01（Ubiquitous）
**The system shall** 在管理后台首页展示数据看板，包含：今日发帖数、今日评论数、今日活跃用户数（独立 IP）、累计帖子数、待处理举报数。

#### FR-13-02（Ubiquitous）
**The system shall** 支持查看近 7 天/30 天的发帖趋势图和互动趋势图。

```sql
-- 帖子表
CREATE TABLE posts (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    content     TEXT NOT NULL,

    image_url   VARCHAR(500),
    delete_hash VARCHAR(255) COMMENT '删除密码的 BCrypt 哈希，可为 NULL（用户未设密码）',

    FULLTEXT INDEX ft_content (content)

-- 评论表

    id          BIGINT AUTO_INCREMENT PRIMARY KEY,

    FOREIGN KEY (post_id) REFERENCES posts(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

    id          BIGINT AUTO_INCREMENT PRIMARY KEY,

    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 管理员表

    id          BIGINT AUTO_INCREMENT PRIMARY KEY,

    id          BIGINT AUTO_INCREMENT PRIMARY KEY,

    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- IP 封禁表（评审新增）
CREATE TABLE ip_blocks (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    ip_hash     VARCHAR(64) NOT NULL UNIQUE COMMENT 'IP 的 SHA-256 哈希',
    reason      VARCHAR(200) COMMENT '封禁原因',
    blocked_by  BIGINT COMMENT '操作的管理员 ID',
    blocked_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (blocked_by) REFERENCES admins(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```
