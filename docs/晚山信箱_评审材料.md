## 二、产品视角 — 需求解释与待确认项

### 2.1 需求核心要点复述

- **零门槛**：无需注册/登录，打开浏览器即可发帖和浏览
- **纯匿名**：系统自动分配随机昵称（如"安静的考拉"），同一设备保持昵称不变
- **先发后审**：内容发布立即可见，管理员后台巡检处理违规
- **社区自治+管理**：用户可举报 + 管理员后台处理 + 敏感词自动拦截
- **图片可选**：MinIO 自部署，不上传则纯文字发布

### 2.2 产品侧待确认问题

| # | 问题 | 影响范围 | 建议决策 | 确认方 |
|---|------|---------|---------|--------|
| P-01 | 删除密码是否必填？PRD 目前设为可选。如果用户不发密码，违规帖子管理员删了就行，但用户自身无法删除 | 发帖流程、用户隐私控制 | **建议可选**：大部分用户懒得设密码，管理员兜底删除违规 | 产品 |
| P-02 | 标签是否支持多选？PRD 未明确单标签还是多标签 | 发帖 UI、数据库设计 | **建议单标签**：降低选择成本，V2 再考虑多标签 | 产品 |
| P-03 | 管理员是否需要多角色（超级管理员/内容审核员）？PRD 目前只有一种 admin 角色 | 管理后台、权限设计 | **建议 V1 单一角色**：团队小，没必要拆分 | 产品 |
| P-04 | 首页排序"最热"的计算公式？点赞+评论+抱抱+同感 加权？还是仅点赞？ | 内容分发 | **建议**：热度 = 点赞×2 + 评论×3 + 抱抱×1 + 同感×1，按 24 小时内热度排序 | 产品+后端 |
| P-05 | 是否需要帖子置顶功能（管理员）？ | 管理后台 | **建议 V1 不加**：先验证基础功能 | 产品 |
| P-06 | Cookie 清除后匿名昵称会变，用户能否接受？ | 用户体验 | **建议接受**：这正是匿名的体现。可在发帖时提示"当前昵称：安静的考拉"增强认知 | 产品 |

## 四、研发视角 — 技术评审与依赖识别

### 4.1 后端技术关键问题

| # | 问题 | 分析 | 建议 | 确认方 |
|---|------|------|------|--------|
| B-01 | MySQL 全文索引对中文支持如何？ | MySQL 8.0 的 ngram parser 支持中文分词，但精度不如 ElasticSearch | **V1 可用**：配置 `ngram_token_size=2`，数据量 < 10w 时性能 OK | 后端 |
| B-02 | 敏感词 Trie 树的并发安全？ | 管理后台修改词库后需要更新内存中的 Trie 树 | **方案**：用 `ReadWriteLock` 保护，或直接用 `ConcurrentHashMap` 构建简化版 | 后端 |
| B-03 | 图片上传是否需要鉴权？ | 匿名用户无登录态，无法用 JWT 鉴权 | **方案**：前端获取临时上传 Token（调用 `/api/v1/upload/token`），5 分钟有效，IP 绑定 | 后端 |
| B-04 | 点赞去重的 Redis Key 设计？ | IP+Cookie 组合作为去重标识 | **方案**：Key=`like:{type}:{target_id}:{ip_hash}:{cookie_id}`，TTL=7天，同步回 MySQL 计数 | 后端 |
| B-05 | 管理后台 JWT Token 过期策略？ | 安全 vs 体验的平衡 | **建议**：Access Token 2h + Refresh Token 7d，后台操作较少可接受 | 后端 |
| B-06 | 帖子软删除后数据保留多久？ | GDPR/个保法合规考虑 | **建议**：软删除保留 30 天后物理删除（定时任务），仅保留统计计数 | 后端+产品 |

### 4.2 前端技术关键问题

| # | 问题 | 分析 | 建议 | 确认方 |
|---|------|------|------|--------|
| F-01 | C 端和管理后台是否共用组件库？ | 两个应用定位不同，但基础组件可复用 | **建议分两个独立 entry**，共享一个 `packages/ui` 基础组件包 | 前端 |
| F-02 | 图片上传前的压缩方案？ | 需在浏览器端压缩 | **方案**：用 Canvas API 压缩，限制宽度 1200px，质量 0.8，输出 WebP | 前端 |
| F-03 | 前端路由模式？Hash vs History？ | History 需要 Nginx 配合，Hash 简单但有 # 号 | **建议 History 模式**：Nginx 配置 `try_files` 即可 | 前端 |
| F-04 | 是否需要 SSR/SSG？ | 树洞内容偏动态，SEO 需求不强 | **建议 V1 纯 CSR**：不引入 Nuxt.js，降低复杂度 | 前端 |

### 4.3 运维与部署关键问题

| # | 问题 | 分析 | 建议 | 确认方 |
|---|------|------|------|--------|
| O-01 | MinIO 数据备份策略？ | 单节点无高可用 | **V1 接受单点风险**，定期手动备份 `/data/minio` 目录到本地 | 运维 |
| O-02 | MySQL 备份策略？ | 数据丢失风险 | **建议**：每日凌晨自动 `mysqldump` + 保留最近 7 天 | 运维 |
| O-03 | 是否需要 CI/CD？ | V1 团队小，手动部署可接受 | **建议**：GitHub Actions 自动构建 Docker 镜像，手动触发部署 | 运维 |
| O-04 | 域名和 SSL 证书方案？ | HTTPS 是基本要求 | **建议**：Let's Encrypt + acme.sh 自动续期，Nginx 配置 SSL 终端 | 运维 |
